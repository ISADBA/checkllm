# LLM 模型真伪评测深度调研

更新时间：2026-04-20

## 1. 调研目标

这份调研聚焦的问题不是“怎么做通用模型评测”，而是更具体的：

**如果要做 LLM 模型的真伪评测，可以从哪些维度展开？**

这里的“真伪评测”不是单纯问几个问题看回答像不像，而是要系统回答以下几类问题：

1. 这个 endpoint 声称自己是某个官方模型，它到底是不是？
2. 它是否只是同一家族但被低配模型替代？
3. 它是否经过 relay 包装、协议伪装、输出重写、区域切换或能力掺水？

因此，LLM 真伪评测本质上不是一个单一评分任务，而是一套**多层证据联合判断系统**。

## 2. 真伪评测的三个层次

做平台设计前，最好先把“真伪”拆成三类问题。

### 2.1 身份真实性

它回答的是：

- 这个接口声称自己是 `Claude Sonnet 4.6`
- 或 `GPT-5.4`
- 或 `Gemini`

它到底是不是这个声明中的模型身份。

### 2.2 代际真实性

它回答的是：

- 这个接口可能确实来自同一家模型供应商
- 但它是否用低一级模型冒充高一级模型
- 或者用旧版本模型冒充新版本模型

### 2.3 路径真实性

它回答的是：

- 结果是否经过 relay 中间层重写
- 是否存在多模型路由
- 是否不同地区走不同上游
- 是否有缓存污染
- 是否有后处理层篡改输出

如果未来做企业级平台，这三层都必须覆盖。

## 3. 研究现状带来的重要启发

最近最直接相关的一类工作，不是在讨论抽象的模型归属理论，而是在审计真实商业 API。

其中最值得关注的一篇是：

- [Real Money, Fake Models: Deceptive Model Claims in Shadow APIs](https://huggingface.co/papers/2603.01919)

这项工作的重要启发是：

- 真实商业 shadow API 中确实存在虚假模型声明问题
- 问题不只是回答质量下降
- 还包括 utility、安全行为和模型验证三方面失真

这意味着：

**现实世界里的模型真伪，不是单纯的文本相似性问题，而是协议、行为、能力、安全和运行路径的联合失真问题。**

## 4. 真伪评测的核心结论

在调研现有研究和工程实践后，可以得到一个非常明确的结论：

### 4.1 单维度判断不可靠

无论是：

- 只看回答内容
- 只看 benchmark
- 只看延迟
- 只看协议结构

都不能单独证明真伪。

### 4.2 需要多层证据联合判断

一个真正可用的真伪评测框架，至少要覆盖：

1. 来源证明维度
2. 协议与接口指纹维度
3. 行为指纹维度
4. 能力画像维度
5. 统计一致性维度
6. 安全行为一致性维度
7. 水印与归属维度
8. 运行遥测与侧信道维度
9. 多地域与多路径一致性维度
10. 长期漂移与对抗鲁棒性维度

## 5. 来源证明维度

这是理论上证据最强的一层。

### 5.1 它回答什么

它试图回答：

- 这个模型是否来自声明中的官方提供方
- 是否来自受信执行环境
- 是否具有可验证来源证明

### 5.2 可以检查什么

- 厂商签名
- 推理 attestation
- 可信执行环境证明
- 官方 model id 与 version 映射
- 供应商侧可验证 provenance

### 5.3 价值

如果厂商提供强 attestation，这一层的证据强度会远高于黑盒行为猜测。

### 5.4 局限

现实情况是：

- 大多数商业 API 目前没有对外提供强可验证推理 attestation
- 企业客户通常只能看到 HTTP 接口，而不是底层执行证明

### 5.5 对平台的意义

即使短期做不到，也建议在系统设计中预留：

- `provenance_evidence`
- `attestation_status`
- `provider_verification_mode`

把它作为未来增强能力，而不是完全忽略。

## 6. 协议与接口指纹维度

这是最适合早期工程化落地的一层。

### 6.1 它回答什么

它回答的是：

- 这个 endpoint 在协议层面像不像它声称的那家服务

### 6.2 可以检查什么

- 请求路径与 schema
- 流式事件生命周期
- usage 字段结构
- stop reason / finish reason
- tool call 格式
- role / content 结构
- 错误码与错误体风格
- 流式 chunk 粒度和事件顺序

### 6.3 参考官方协议

这一层必须优先参考官方文档而不是二手经验：

- [Anthropic Messages API Examples](https://docs.anthropic.com/en/api/messages-examples)
- [Anthropic Streaming Messages](https://docs.anthropic.com/claude/reference/messages-streaming)
- [OpenAI Streaming Responses](https://platform.openai.com/docs/guides/streaming-responses)
- [OpenAI Responses Streaming Reference](https://platform.openai.com/docs/api-reference/responses/streaming)

### 6.4 价值

很多 relay 可以伪造最终文本，但很难完整伪造：

- 所有流式事件细节
- usage 计数习惯
- tool 调用格式
- 错误语义

### 6.5 局限

协议一致并不能单独证明模型真实性，只能说明“接口长得像”。

### 6.6 建议拆分的子指标

- schema conformity
- stream lifecycle conformity
- usage consistency
- tool-call conformity
- error semantics conformity

## 7. 行为指纹维度

这是黑盒识别里最重要的一层之一。

### 7.1 它回答什么

它回答的是：

- 这个模型的行为风格是否稳定地像目标模型或模型家族

### 7.2 研究启发

这方面可以重点参考：

- [TRAP: Targeted Random Adversarial Prompt Honeypot for Black-Box Identification](https://huggingface.co/papers/2402.12991)
- [Hide and Seek: Fingerprinting Large Language Models with Evolutionary Learning](https://huggingface.co/papers/2408.02871)

这些工作的关键启发是：

- 可以设计针对性的 honeypot prompts
- 可以自动搜索最有区分度的提示词
- 黑盒条件下依然可以构建指纹

### 7.3 可以检查什么

- 特殊 prompt 的响应模式
- 多轮记忆衔接方式
- refusal 模板
- reasoning 暴露习惯
- 格式偏好
- 工具规划风格
- 边界输入时的稳定风格

### 7.4 价值

这层是“黑盒身份识别”的核心。

### 7.5 局限

- 容易受版本升级影响
- 容易受 wrapper 干扰
- 单个 prompt 不可靠，必须使用 prompt set

### 7.6 建议拆分的子指标

- family fingerprint score
- honeypot hit rate
- refusal signature similarity
- multi-turn memory signature
- planning pattern similarity

## 8. 能力画像维度

这一层主要用于识别“同家族低配冒充高配”。

### 8.1 它回答什么

它回答的是：

- 虽然这个模型看起来属于目标家族
- 但它的能力层级是否与声明模型一致

### 8.2 可以检查什么

- 长链推理
- 长上下文跟踪
- 编码修复
- 多约束结构化输出
- 工具调用规划
- 多跳推理
- 多语言稳定性

### 8.3 价值

现实里很多“假货”不是完全无关模型，而是：

- 同系列低档版本
- 旧版本
- 非 reasoning 变体
- 便宜模型加后处理

### 8.4 局限

能力受以下因素影响很大：

- 温度
- 系统提示
- 上游包装
- 安全策略

所以能力不能单独作为身份判定依据。

### 8.5 建议拆分的子指标

- claimed-vs-observed capability gap
- reasoning tier fit
- code task parity
- long-context parity

## 9. 统计一致性维度

这一层很重要，但经常被低估。

### 9.1 它回答什么

它回答的是：

- 在同样输入分布下，这个 endpoint 的输出分布是否与声明模型一致

### 9.2 可以检查什么

- 同 prompt 多次采样分布
- 语义聚类分布
- benchmark 分布差异
- 多任务输出风格统计
- 不同接口之间的分布偏移

### 9.3 研究启发

`Real Money, Fake Models` 这类工作表明，商业 shadow API 在分布层面会明显偏离官方模型。

来源：

- [Real Money, Fake Models](https://huggingface.co/papers/2603.01919)

### 9.4 价值

有些 API 能模仿几个固定问题，但在分布层面会暴露差异。

### 9.5 局限

这一层需要：

- 足够样本量
- 稳定任务集
- 比较严格的实验控制

### 9.6 建议拆分的子指标

- distributional similarity
- benchmark parity
- repeated-sampling variance
- semantic drift score

## 10. 安全行为一致性维度

这一层经常被忽略，但对真实性判断非常重要。

### 10.1 它回答什么

它回答的是：

- 模型在安全边界、拒答行为和越狱响应上，是否与声明模型一致

### 10.2 为什么重要

真实商业模型通常有比较稳定的安全边界。  
而 shadow API 或低成本替代模型，往往在这一层偏差明显。

`Real Money, Fake Models` 也明确发现 shadow APIs 在 safety 行为上存在明显偏离。

来源：

- [Real Money, Fake Models](https://huggingface.co/papers/2603.01919)

### 10.3 可以检查什么

- jailbreak 成功率
- harmful prompt 拒答率
- policy wording
- 编码绕过响应
- tool misuse 拒绝模式
- 边缘安全场景表现

### 10.4 局限

这一层可能被上层安全代理或 wrapper 干扰，所以它既可能表示“模型不是那个模型”，也可能表示“路径上有额外安全层”。

### 10.5 建议拆分的子指标

- refusal consistency
- safety boundary alignment
- unsafe completion divergence
- wrapper safety override likelihood

## 11. 水印与归属维度

如果未来能和厂商合作，这一层会成为强增强证据。

### 11.1 它回答什么

它回答的是：

- 输出中是否存在可验证的 watermark 或 ownership signal

### 11.2 相关研究

这方面可以参考：

- [Watermarking for Large Language Models: A Survey](https://www.mdpi.com/2227-7390/13/9/1420)
- [Watermarking techniques for large language models: a survey](https://link.springer.com/article/10.1007/s10462-025-11474-6)
- [A Semantic Invariant Robust Watermark for Large Language Models](https://openreview.net/forum?id=6p8lpe4MNf)

### 11.3 可以检查什么

- 文本水印可检出率
- paraphrase 后的水印保留率
- 同义替换后的鲁棒性
- 误报率
- 漏报率

### 11.4 价值

如果某家官方模型有对外可验证 watermark，这会是非常强的身份证据。

### 11.5 局限

现实中仍然存在几个问题：

- 多数商业 API 未公开 watermark 验证接口
- paraphrase 和 relay 重写会破坏信号
- 误报和漏报控制难度高

### 11.6 结论

这层更适合作为平台中后期增强能力，而不是 MVP 唯一核心。

## 12. 运行遥测与侧信道维度

这一层对企业级平台非常重要。

### 12.1 它回答什么

它回答的是：

- 这个模型或路径在运行方式上，是否像它声称的服务

### 12.2 可以检查什么

- first token latency
- total latency
- tokens/s
- chunk cadence
- usage 计数习惯
- cache hit 模式
- 输出长度分布
- 成本/性能关系

### 12.3 为什么重要

很多假货文本上能装，但在这些地方很难完全装得一致：

- 响应速度曲线
- 流式节奏
- 计费/usage 风格
- cache 行为

### 12.4 旁证

OpenAI 文档甚至明确提到过 streaming delta 的某些处理与 side-channel 风险有关，这反向说明流式和时序特征本身确实会泄漏路径信息。

来源：

- [OpenAI Responses Streaming Reference](https://platform.openai.com/docs/api-reference/responses/streaming)

### 12.5 局限

这层只能作为辅助证据，不能单独认定真伪。

### 12.6 建议拆分的子指标

- latency signature
- throughput signature
- stream cadence signature
- cache behavior signature
- cost/perf anomaly score

## 13. 多地域与多路径一致性维度

这一层对未来平台差异化非常关键。

### 13.1 它回答什么

它回答的是：

- 同一个 endpoint 在不同地区、不同网络路径下，返回的是不是同一个模型行为

### 13.2 可以检查什么

- 东京 vs 新加坡 vs 法兰克福 vs 美西
- 公网路径 vs 私有专线
- 不同时段
- 不同租户
- 不同 key 分组

### 13.3 为什么重要

真实商业 relay 问题往往不是“全局固定造假”，而是：

- 某些 region 偷偷切便宜模型
- 某些时段降级
- 某些 key 命中不同上游
- 海外与国内走不同路径

### 13.4 价值

这不仅是真实性判断的重要维度，也会成为企业级 SLA 平台的核心能力。

### 13.5 建议拆分的子指标

- cross-region parity
- cross-network parity
- tenant/path consistency
- time-window consistency

## 14. 长期漂移维度

真伪评测不能只做一次性测试。

### 14.1 它回答什么

它回答的是：

- 今天是真的，明天还是真的吗
- 一个月后它是否仍然与声明模型一致

### 14.2 可以检查什么

- 周期性行为指纹稳定性
- benchmark 趋势变化
- refusal 模板变化
- 延迟分布变化
- 不同地域差异的长期漂移

### 14.3 为什么重要

企业真正关心的是：

- 高峰期是否偷偷降级
- 某个供应商是否逐步偏离承诺
- 某个 relay 是否在不同时间切换模型

### 14.4 建议拆分的子指标

- fingerprint drift
- capability regression
- safety drift
- routing drift
- regional drift

## 15. 对抗鲁棒性维度

没有这层，你的评测体系很容易被绕过。

### 15.1 它回答什么

它回答的是：

- 这些评测维度在 paraphrase、温度变化、语言变化和 wrapper 包装下是否依然有效

### 15.2 可以检查什么

- paraphrase robustness
- temperature robustness
- multilingual robustness
- system wrapper robustness
- long-context robustness
- adversarial suffix robustness

### 15.3 研究启发

TRAP 和 watermarking 相关研究都强调：  
如果 fingerprint 对轻微改写就失效，它就不适合真实商业环境。

来源：

- [TRAP](https://huggingface.co/papers/2402.12991)
- [A Semantic Invariant Robust Watermark for Large Language Models](https://openreview.net/forum?id=6p8lpe4MNf)

## 16. 这些维度的证据强度排序

如果把这些维度按证据强度分层，大致可以分成三组。

### 16.1 强证据层

- 来源证明 / attestation
- watermark / ownership verification
- 协议与流式事件指纹

### 16.2 中强证据层

- 行为指纹
- 统计一致性
- 多地域一致性
- 长期漂移

### 16.3 辅助证据层

- 能力画像
- 安全行为一致性
- 运行遥测与侧信道

原因是：

- 能力强不代表身份真
- 安全像不代表身份真
- 延迟像也不代表身份真

但当多个层次的证据一致时，置信度会显著提高。

## 17. 适合工程落地的评分框架

如果未来要做真实性引擎，建议不要输出一个简单的“真假分数”，而是至少拆成四个分。

### 17.1 Provenance Score

来源可信度。主要看：

- attestation
- watermark
- 官方标识
- 协议一致性

### 17.2 Identity Score

身份可信度。主要看：

- 行为指纹
- family fingerprint
- honeypot prompt 命中情况
- 分布一致性

### 17.3 Fidelity Score

保真度。主要看：

- 能力是否接近声明模型
- 质量是否稳定
- 安全边界是否一致

### 17.4 Path Integrity Score

路径完整性。主要看：

- relay 改写痕迹
- 多地域差异
- 缓存异常
- wrapper 干扰
- 多时段一致性

这样比输出一个总分更可解释。

## 18. MVP 阶段最值得先做的维度

如果现在就开始做产品，不建议一次把所有层都做满。

### 18.1 第一阶段建议优先做

- 协议与接口指纹
- 行为指纹
- 统计一致性
- 多地域一致性
- 长期漂移

这是最现实、最接近商业问题、也最容易形成差异化的一组。

### 18.2 第二阶段建议补充

- 安全行为一致性
- 能力画像
- 运行遥测与侧信道

### 18.3 第三阶段建议补充

- watermark / ownership verification
- provenance attestation

这些能力通常需要：

- 厂商合作
- 更成熟的协议标准
- 更强基础设施支持

## 19. 最终结论

如果要做 **LLM 模型真伪评测**，最合理的展开维度不是“多问几道题”，而应该基于以下 10 类维度建立联合证据体系：

1. 来源证明维度
2. 协议与接口指纹维度
3. 行为指纹维度
4. 能力画像维度
5. 统计一致性维度
6. 安全行为一致性维度
7. 水印与归属维度
8. 运行遥测与侧信道维度
9. 多地域与多路径一致性维度
10. 长期漂移与对抗鲁棒性维度

这套框架最重要的核心思想是：

**模型真伪不是单点判定，而是多层证据叠加后的置信结论。**

如果后续要把它做成平台，那么最合理的系统形态应该是：

- `Authenticity Engine`
- `Quality Evaluation Engine`
- `Monitoring & Probe Network`

三层并行，而不是做成一个简单的 prompt-based 检测脚本。

## 20. 从“研究”走向“校验引擎”时必须先回答的问题

如果你的目标不是写一篇调研，而是做一个真正可交付的“模型真伪校验引擎”，那么首先要回答的不是“能不能百分之百识别”，而是：

1. 哪些问题可以高置信识别？
2. 哪些问题只能做风险提示？
3. 哪些问题必须依赖官方合作或额外遥测？

这一点非常重要，因为“模型真伪”在工程上并不是单一命题，而是至少包含以下 6 类可审计对象：

1. `身份冒充`
   - 声称是 `GPT-5.4`，实际上不是这个模型或甚至不是这个厂商
2. `代际降级`
   - 声称是同品牌高阶模型，实际上是低阶、旧版或廉价变体
3. `路径篡改`
   - 输出经过 relay、二次改写、缓存拼接、后处理或多上游路由
4. `计费失真`
   - usage 口径异常、token 计数不准、返回的 token 信息不符合预期
5. `稳定性失真`
   - 某些时段、某些地区、某些 key 被切到不同上游
6. `策略污染`
   - 中转平台注入自带 system prompt、安全代理或输出改写规则

只有把目标拆成这 6 类，你的引擎才能既可解释，又能落地。

## 21. 可行性边界：哪些能“证明”，哪些只能“推断”

对于模型真伪校验，工程上必须接受一个现实：

### 21.1 可以较强证明的

- 协议不一致
- 官方接口风格不一致
- usage 计数与官方 tokenizer 或官方 usage 回包长期偏离
- 流式事件序列、错误语义、tool schema 与官方不一致
- 多地域/多时段出现显著路径分裂
- 长期行为指纹与官方基线稳定偏离

这些问题虽然未必能“证明底层真实模型是谁”，但通常足以证明：

- 它不是官方原生直连体验
- 或至少不是“声明中的那种官方服务形态”

### 21.2 只能高概率推断的

- 同品牌高配是否被低配替代
- 是否被同家族旧版模型替换
- 是否存在轻量 wrapper 改写
- 是否有隐藏 system prompt 影响输出

这类问题通常需要：

- prompt set
- 对照基线
- 多次重复采样
- 多维证据联合

### 21.3 很难单独黑盒证明的

- 底层权重是否就是官方那一版
- 是否经过轻微蒸馏但能力高度接近
- 是否存在厂商内部未公开版本切换

因此，产品定义上不要承诺“绝对验明正身”，而应承诺：

**输出可解释的真实性风险结论，以及针对业务损失的可操作告警。**

## 22. 面向真实业务风险的威胁模型

如果用户是通过三方平台接入模型，你最需要防的不是抽象学术问题，而是下列真实作弊路径。

### 22.1 不同品牌模型灌水

典型模式：

- 声称是 OpenAI，实际上转到其他开源或闭源模型
- 使用一个便宜模型，加上后处理模仿目标口吻

主要检测维度：

- 协议与接口指纹
- 行为指纹
- refusal/safety 模式
- stream cadence
- usage 与 tokenizer 一致性

### 22.2 相同品牌模型关税

这里的“关税”本质上是：

- 同品牌低阶模型冒充高阶模型
- 同品牌旧版模型冒充新版模型
- reasoning 变体被 non-reasoning 变体替代

主要检测维度：

- 能力画像
- 难例行为指纹
- 长上下文和工具规划能力
- 重复采样分布
- 长期漂移

### 22.3 乱扣费

典型模式：

- 输入输出 token 虚报
- cache 命中不透明
- 流式中途中断但 usage 返回异常
- 返回的 usage token 信息与输入输出规模不匹配

主要检测维度：

- 本地 tokenizer 对账
- 官方 usage 对账
- chunk 数量与最终 usage 对账
- 输入长度、输出长度、usage 三者关系校验

### 22.4 稳定性掺水

典型模式：

- 高峰期切廉价模型
- 某些 region/tenant 单独降级
- 失败重试后命中不同上游

主要检测维度：

- 多地域探针
- 多时段基线比较
- key/tenant 分组对照
- 首 token 延迟和 tokens/s 漂移
- 返回风格与能力漂移联动分析

### 22.5 自带 prompt 污染

典型模式：

- 中转层增加隐藏 system prompt
- 附加品牌宣传、敏感词拦截、固定免责声明
- 强行改写工具调用或输出格式

主要检测维度：

- prompt echo / hidden instruction probe
- 指令优先级冲突测试
- 结构化输出约束测试
- refusal wording 偏移
- tool planning 改写检测

## 23. 把“真实性”拆成五类可交付结论

如果最终产品只输出一个“真假分数”，用户很难采取行动。更合理的做法是输出五类结论。

### 23.1 Official Compatibility

它回答：

- 这个 endpoint 的协议、usage、流式和错误语义是否兼容官方

### 23.2 Identity Confidence

它回答：

- 这个 endpoint 是否高概率就是声明中的模型身份

### 23.3 Tier Fidelity

它回答：

- 即使属于同品牌，它是否仍然符合声明中的能力档位

### 23.4 Usage Integrity

它回答：

- token 统计与 usage 回包是否一致、是否符合预期

### 23.5 Route Integrity

它回答：

- 是否存在中转改写、区域切换、缓存污染、租户差异或时段降级

这五类结论与真实业务损失是直接对应的，因此比单一 authenticity score 更有产品价值。

## 24. 针对“隐藏 prompt / 包装层”的专门鉴定策略

这是你这个方向里很值得深挖的一块，因为很多三方平台未必换了模型，但会偷偷加一层系统策略。

### 24.1 为什么它值得单列

隐藏 system prompt 不一定改变基础能力，但会显著改变：

- 拒答边界
- 品牌自称
- 输出格式
- 工具调用习惯
- 多轮指令服从顺序

换句话说：

**用户买到的已经不是官方原始模型行为，而是“官方模型 + 中间层策略”。**

### 24.2 可以如何测

建议设计 4 类 probe：

1. `优先级冲突 probe`
   - 在 system/user/developer 风格约束冲突时，看模型遵循哪一层
2. `品牌自述 probe`
   - 观察其自我归因、限制说明、免责声明模板是否异常一致
3. `格式约束 probe`
   - 测试 JSON、XML、tool call、markdown 等输出是否被额外重写
4. `反向泄漏 probe`
   - 设计低风险提示词，观察是否暴露固定前缀、安全术语或平台标记

### 24.3 需要注意什么

- 不要依赖直接问“你的 system prompt 是什么”
- 要依赖多轮、冲突、变体和对照实验
- 要区分“模型本身的官方安全策略”和“第三方附加包装层”

## 25. token 与计费校验的可行方案

对于商业用户，这部分往往比“模型到底是不是”还更刚需。

### 25.1 为什么可做

因为 token 与 usage 问题比身份问题更容易形成强证据：

- 输入是你已知的
- 输出是你收到的
- tokenizer 是可重复计算的
- 官方 usage 风格通常相对稳定

### 25.2 建议拆成两层校验

#### A. 静态计数校验

比较：

- 本地 tokenizer 估算值
- 对方 usage 回包值
- 官方基线接口的 usage 风格

核心看：

- 输入 token 偏差率
- 输出 token 偏差率
- 总 token 偏差率

#### B. 流式 usage 校验

重点看：

- stream 中断前后的 usage 返回逻辑
- tool call / reasoning 摘要是否引入异常 usage
- 空响应、截断响应是否返回异常 usage

### 25.3 风险判定建议

建议至少定义 3 类异常：

- `计数异常`
- `usage 返回异常`
- `流式 usage 异常`

其中 `usage 返回异常` 的商业价值通常最高。

## 26. “同品牌降级冒充”应该怎么测

这类问题最难，因为协议常常完全一致。

### 26.1 核心思路

不要做通用 benchmark 堆题，而要做：

- `档位分离任务`
- `边界稳定性任务`
- `高成本能力任务`

### 26.2 什么叫档位分离任务

就是那些能明显拉开高配与低配模型差距，但不容易被模板化作弊的任务，例如：

- 长约束代码修复
- 长上下文定位与引用
- 多工具规划
- 多轮状态保持
- 复杂结构化抽取
- 指令冲突下的稳定执行

### 26.3 评价方法

不要只看均分，而要看：

- 通过率
- 边界 case 崩溃率
- 重试后恢复率
- 多次采样方差
- 长短任务性能斜率

如果一个 endpoint 在简单任务上接近官方，但在高成本任务上显著塌陷，通常就有很强的降级嫌疑。

## 27. 探针数据集应该怎么设计

如果你要做成引擎，数据集设计比模型评分公式更关键。

### 27.1 四层任务池

建议将 probe corpus 拆成四层：

1. `协议层任务`
   - schema、stream、tool、error、usage
2. `指纹层任务`
   - honeypot、refusal、格式偏好、多轮衔接
3. `能力层任务`
   - tier 区分、代码、长上下文、规划
4. `业务层任务`
   - usage、稳定性、跨地域

### 27.2 数据集构造原则

- 每类任务必须有变体，避免被记忆
- 每类任务必须可重放，方便长期比较
- 每类任务必须可解释，方便用户看报告
- 每类任务必须可做官方基线对照

### 27.3 数据集更新策略

建议至少分三套：

- `固定锚点集`
  - 用于长期趋势比较
- `滚动挑战集`
  - 用于防止被对手适配
- `客户定制集`
  - 用于贴近特定业务场景

## 28. 校验引擎的系统架构建议

如果把它真正做成产品，建议至少拆成以下模块。

### 28.1 Probe Orchestrator

负责：

- 任务编排
- 多地域调度
- 多时段调度
- 多 key / 多 tenant 对照
- 重试与实验控制

### 28.2 Baseline Registry

负责维护：

- 官方模型基线
- 版本窗口
- tokenizer 与 usage 规则
- 协议 schema
- 历史漂移记录

### 28.3 Evidence Extractor

负责从每次调用中提取：

- 文本行为特征
- stream 时序特征
- usage 特征
- error 特征
- 工具调用特征

### 28.4 Scoring & Adjudication

负责：

- 多维指标归一化
- 风险加权
- 规则与统计模型联合判定
- 输出可解释结论

### 28.5 Usage Auditor

负责：

- tokenizer 对账
- usage 对账
- usage 单调性检查
- 流式 usage 一致性检查

### 28.6 Drift Monitor

负责：

- 日级、周级、月级漂移检测
- 区域漂移
- 租户漂移
- 高峰期降级识别

## 29. 判定逻辑不要只靠模型分数，要同时有规则引擎

这是一个很关键的工程决策。

### 29.1 为什么

有些异常根本不需要 ML 判定：

- 官方 schema 不匹配
- usage 字段缺失或口径异常
- 流式生命周期不符合官方
- usage 数值与输入输出规模不匹配

这些都应直接由规则引擎给出硬告警。

### 29.2 更合理的判定方式

建议使用：

- `规则硬判定`
- `统计偏差判定`
- `黑盒指纹相似度判定`

三层并行。

其中：

- 规则硬判定负责高精度低召回
- 统计偏差负责长期风险发现
- 指纹相似度负责身份和代际推断

## 30. 输出报告应该长什么样

报告结构会直接影响产品价值。

### 30.1 建议的主结论

- `结论`
  - 高置信官方一致 / 高风险疑似替代 / 存在中转篡改 / 存在计费异常 / 存在区域降级
- `置信度`
  - 高 / 中 / 低
- `影响面`
  - 身份 / 质量 / 计费 / 稳定性 / 安全边界

### 30.2 建议的证据卡片

每条证据都给出：

- 证据类型
- 观测值
- 官方基线
- 偏差程度
- 可能解释
- 建议动作

例如：

- `usage 偏差`
  - 连续 500 个请求中，输出 token 回包比本地 tokenizer 高 18%
- `跨地域漂移`
  - 美西与新加坡在相同挑战集上通过率相差 27%
- `包装层嫌疑`
  - 结构化输出任务中持续注入固定免责声明模板

## 31. MVP 最小闭环应该怎么做

如果你现在要在 `checkllm` 里先做第一版，最合理的 MVP 不是“全鉴定平台”，而是：

### 31.1 第一阶段

先把最能打业务价值的三件事做出来：

1. `协议/usage/stream 官方一致性校验`
2. `同品牌降级风险识别`
3. `usage token 合理性审计`

### 31.2 为什么是这三件

因为它们分别对应最直接的客户痛点：

- 你到底是不是官方兼容
- 你是不是拿低配冒充高配
- 你有没有多扣钱

### 31.3 第一阶段不必追求的

暂时不必把以下能力做到很深：

- watermark
- provenance attestation
- 全自动模型家族归因
- 非常重的通用 benchmark 平台

先把“能抓住商业作弊”的部分做扎实，产品就已经有价值。

## 32. 最终建议：把目标定义成“真实性风控”而不是“真假识别”

从产品战略上看，我更建议你把这个方向定义成：

- `LLM Authenticity Risk Engine`
- 或 `Model Fidelity & Usage Audit Engine`

而不是一个过于绝对的“真假识别器”。

原因很简单：

1. 黑盒条件下很难做绝对身份证明
2. 企业真正愿意付费的是风险发现、usage 审计和供应商约束
3. “真实性风控”天然可以覆盖你最关心的几类问题：
   - 品牌灌水
   - 同牌降级
   - usage 异常
   - 稳定性漂移
   - 隐藏 prompt 污染

所以更准确的产品表述应该是：

**我们不是试图神奇地读出底层权重，而是用多层证据去验证：你买到的服务，是否真的等于你以为你买到的那个模型服务。**
